# 1.模块规划

User-Role Permissions：是账号与角色关联授权后的结果，选择User-Role即选择账号的系统权限。

User Account：来源于Accounts Management，系公司内部工作人员的账号。

System Role：通过各项配置，最终形成的数据与操作权限合集。

Data Permissions：数据权限，基于Role的组织层级与汇报关系，而确定的对于系统Entities的可见范围。

Operating Permissions：操作权限，配置的角色在各模块的具体操作权限。

## 1）本期原则

本期主要考虑到完全版的权限管理模块工作量，以及当前系统业务的不稳定风险，所以采用简化的松散式账号授权。

本期并不能从系统层面完全杜绝操作权限的重叠、冲突，亦不能数据权限的精准管理。

需要通过手动管理，以实现基础的权限管理，支撑业务开展。

## 2）后期规划

后期会根据系统业务的稳定程度，逐步将系统角色固定化。

基于系统角色，后续也会对数据可见性进行精准管控，修补组织层级与汇报关系不能解决的数据管理问题。

基于系统角色，后续也会对具体的操作权限配置增补互斥或级联逻辑，以保证权限的不重叠不冲突。

# 3.功能需求

## 1）部门和角色管理

### ①部门创建和编辑

系统管理员通过权限管理模块创建部门时，需要填写部门名称和部门描述。

每个部门被分配一个唯一标识符，以区分不同部门。

管理员可选择父部门，从而形成部门的层次结构。

### ② 角色创建和编辑

在部门下创建角色时，管理员需提供角色名称和角色描述。

每个角色被赋予唯一标识符，以保持角色的唯一性。

## 2）操作权限设置

### ①操作权限定义

在权限管理模块中，系统管理员定义操作权限时需要输入权限标识符、权限名称和权限描述。

操作权限代表系统中可执行的特定操作或功能。

### ②操作权限分配

在角色管理页面，管理员为每个角色分配操作权限。

每个角色可以具备多个操作权限，这些权限定义了角色可以执行的操作。

## 3.）数据权限设置

### ①汇报关系定义

在角色管理页面，管理员可以为角色指定上级角色，以建立汇报关系链。

汇报关系可以在角色创建或编辑时进行配置。

### ②数据范围定义

每个角色的数据权限需指定明确的数据范围，以控制数据的访问。

管理员在角色设置中，选择数据范围类型：自己及下属、当前部门、当前部门及下属部门、全公司。

## 4）角色关联和授权

管理员在用户管理中，将角色分配给特定用户账号。

每个用户账号可以关联一个或多个角色，以获得相应的权限。

## 5）用户角色管理

管理员可以添加、编辑和删除用户账号与角色之间的关联。

用户账号与角色的关联关系可根据实际情况进行调整。

## 6）数据权限控制逻辑

在TMS系统的权限管理模块中，需要实现数据权限的透传功能，确保上级角色可以查看下级角色的数据，但下级角色不能查看上级角色的数据。同时，数据权限透传不得影响部门层级范围的权限。以下是数据权限透传功能的详细需求说明：

### ①数据权限透传原则

上级角色查看下级数据：当存在上下级角色关系时，上级角色可以透传其数据权限给下级角色，以实现数据权限的共享。例如，如果角色A是角色B的上级，则角色A可以查看角色B的数据。

下级角色不可查看上级数据：下级角色无权查看上级角色的数据，透传功能仅适用于上级角色查看下级数据。

部门层级不透传：数据权限透传不得影响部门层级范围的权限，即使存在上下级角色关系。下级角色的部门数据权限仍然受到部门层级的限制。

### ②数据权限透传逻辑

透传规则判断：当存在上下级角色关系时，系统根据透传原则判断是否应该透传数据权限。仅当上级角色的数据权限适用于下级角色时，系统才进行数据权限透传。

数据权限透传计算：数据权限透传时，系统将上级角色的数据权限透传给下级角色，以确保下级角色可以查看其上级角色的数据。

### ③数据权限透传效果

透传优先：系统应确保透传的上级数据权限优先生效，以满足上级角色查看下级数据的需求。

部门数据范围限制：透传数据权限不得影响下级角色的部门数据范围限制，确保数据权限控制的准确性。
